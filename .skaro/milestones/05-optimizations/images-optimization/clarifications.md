# Clarifications: images-optimization

**Статус (AG-IO-03, 2026-03-25):** В репозитории принята стратегия **локальных WebP в `public/images/`** при `output: 'export'` и `images.unoptimized: true`. Ответ на вопрос 1 ниже (picsum) сохранён как исторический контекст уточнений; **обязательным** подход он больше не считается. Актуальное описание — в `spec.md` и `AI_NOTES.md`.

## Question 1
Как обрабатывать отсутствие реальных файлов изображений при сборке (next export)?

*Context:* Поскольку генерация изображений исключена из границ задачи, билд может упасть или отображать битые ссылки, если файлов нет в public/images.

**Options:**
- A) Использовать проверенные плейсхолдеры (например, picsum.photos) с фиксированными размерами
- B) Создать пустые заглушки (1x1 px WebP) в папке public/images с соответствующими именами
- C) Оставить только код с путями, предполагая, что файлы будут добавлены позже вручную

**Answer:**
Использовать проверенные плейсхолдеры (например, picsum.photos) с фиксированными размерами

## Question 2
Какую стратегию приоритетной загрузки (priority) использовать для HeroSection?

*Context:* Для достижения LCP <2.5s важно правильно настроить загрузку первого экрана, особенно на мобильных устройствах.

**Options:**
- A) Принудительный fetchPriority='high' и priority={true} только для главного фонового изображения
- B) Priority для фонового изображения и логотипа в шапке
- C) Использовать стандартный lazy-load для всех элементов, полагаясь на оптимизацию Next.js

**Answer:**
Принудительный fetchPriority='high' и priority={true} только для главного фонового изображения

## Question 3
Какие конкретные размеры (sizes) указать для адаптивных изображений в секциях Cases и ProductLine?

*Context:* Атрибут sizes критичен для того, чтобы браузер не скачивал слишком большие изображения на мобильных устройствах.

**Options:**
- A) Стандартная сетка: (max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw
- B) Фиксированные размеры на основе макета (например, 400px для карточек)
- C) Не указывать sizes, использовать автоматическое поведение next/image

**Answer:**
Стандартная сетка: (max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw

## Question 4
Нужна ли поддержка формата AVIF в дополнение к WebP?

*Context:* AVIF обеспечивает лучшее сжатие, но увеличивает время сборки при статическом экспорте.

**Options:**
- A) Только WebP (согласно ADR-001 и спецификации)
- B) Включить поддержку AVIF и WebP в next.config.js для максимальной оптимизации
- C) Использовать оригинальные форматы (PNG/JPG) без конвертации

**Answer:**
Только WebP (согласно ADR-001 и спецификации)

## Question 5
Как реализовать 'Inline Expand' для кейсов с точки зрения производительности изображений?

*Context:* Если в скрытом блоке кейса много тяжелых изображений, их загрузка может влиять на метрики, если не настроен lazy loading.

**Options:**
- A) Загружать изображения внутри раскрывающегося блока только при его активации (условный рендеринг)
- B) Использовать стандартный loading='lazy', который сработает при раскрытии блока
- C) Загружать все изображения кейсов сразу для мгновенного открытия

**Answer:**
Использовать стандартный loading='lazy', который сработает при раскрытии блока
