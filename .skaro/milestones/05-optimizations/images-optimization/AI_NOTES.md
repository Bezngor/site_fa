# AI_NOTES: images-optimization

Краткая памятка по тому, как в проекте устроены изображения при статическом экспорте Next.js.

## Архитектура

- **Режим:** `output: 'export'` в `next.config.js` — итог в каталоге `out/` после `npm run build`.
- **Изображения:** `images.unoptimized: true` — штатный on-demand Image Optimization сервера Next **не используется** на экспорте. Браузер запрашивает файлы из `public/` как статику (пути вида `/images/...`).
- **Смысл:** Сайт можно отдавать с любого статического хостинга без Image Optimization API. Качество и вес WebP задаются **до** коммита (ручная или внешняя оптимизация).

## Где что лежит

| Назначение | Файлы | Источник пути / данных |
|------------|--------|-------------------------|
| Hero (LCP) | `public/images/hero.webp` | `texts.hero.backgroundSrc`, `backgroundAlt` в `lib/data/texts.ts` |
| Кейсы | `public/images/cases/case1.webp` … `case3.webp` | `lib/data/cases.ts` — `image`, `imageAlt`, `imageWidth`, `imageHeight` |
| Продуктовая линейка | — (нет растра) | Иконки Lucide в `components/sections/ProductCard.tsx` |
| О нас | — (нет фото) | Lucide в `components/sections/AboutSection.tsx` |

## Компоненты

- **Hero:** `next/image`, `fill`, `priority`, `fetchPriority="high"`, overlay для контраста текста.
- **Cases:** `next/image` рендерится **только** при открытом аккордеоне; `loading="lazy"`; контейнер с фиксированным соотношением сторон; `sizes` для адаптива.

## Замена заглушек на прод

1. Подготовить WebP (желательно с разумными пиксельными размерами под макет).
2. Заменить файлы в `public/images/` с теми же именами **или** обновить пути в `texts.ts` / `cases.ts`.
3. При смене размеров файла обновить `imageWidth` / `imageHeight` в кейсах, если они используются для документации или будущих расширений; для текущего `fill` + aspect-container критичен в первую очередь контейнер.

Внешние URL (например picsum) **не** являются обязательными; при их использовании понадобится `images.remotePatterns` в `next.config.js` и осознанное решение по доступности CDN при сборке.

## CWV — ориентир для проверки перед релизом

Не формальный gate в CI, а чеклист:

- **LCP:** главный кандидат — Hero WebP; проверить мобильную сеть в Lighthouse или полевые данные.
- **CLS:** нет ли сдвигов при загрузке Hero и при раскрытии кейсов; контейнеры с предсказуемой геометрией.
- **INP:** не относится напрямую к картинкам, но тяжёлые изображения на слабых устройствах влияют на общее восприятие — держать вес WebP разумным.
