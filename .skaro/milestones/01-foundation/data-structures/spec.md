# Спецификация: data-structures (Обновлено)

## Контекст
Статические TS/JSON файлы обеспечивают типобезопасность и нулевую задержку (ADR-003). Данные хранятся локально и компилируются в статические страницы. Это исключает необходимость в базе данных и внешних API для контента. Все данные жестко закодированы на русском языке без поддержки i18n.

## User Scenarios
1. **Компонент HeroSection:** Импортирует данные из `/lib/data/texts.ts` и рендерит заголовок и подзаголовок.
2. **Компонент CasesSection:** Читает `cases.json` с 3 кейсами для отображения портфолио.
3. **Компонент HowWeWorkSection:** Итерирует по массиву шагов из `texts.ts` для отрисовки этапов процесса.
4. **Компонент ProductLineSection:** Отображает сетку продуктов из плоского массива в `products.ts`.

## Functional Requirements
- **FR-01: texts.ts (Контент секций)** — Создать файл с типизированными объектами для всех 8 секций. Локализация (i18n) не требуется; строки хранятся напрямую как значения ключей (например, `heroTitle: '...'`).
- **FR-02: cases.json (Портфолио)** — Создать массив из 3 кейсов. Каждый кейс содержит: `title`, `description`, `image`, `metrics`.
- **FR-03: products.ts (Линейка продуктов)** — Создать плоский массив продуктов для `ProductLineSection`. Поддержка вложенных категорий не требуется — все продукты отображаются в единой сетке.
- **FR-04: HowWeWork structure** — Данные для секции "Как мы работаем" должны быть представлены в виде массива объектов: `{ step: number, title: string, description: string }`.
- **FR-05: Формат путей** — Все пути к изображениям в JSON/TS файлах должны быть абсолютными от корня: например, `/images/cases/case1.webp`.
- **FR-06: Структура метрик** — Метрики в кейсах должны иметь строго определенную структуру объекта для единообразия UI: `{ label: string, value: string, unit: string }`.

## Non-Functional Requirements
- **NFR-01:** Полная типизация TypeScript (интерфейсы для всех структур данных в `/types/index.ts`).
- **NFR-02:** Использование путей к изображениям исключительно в формате WebP для оптимизации.
- **NFR-03:** Все текстовые данные должны быть на русском языке.
- **NFR-04:** Неизменяемость данных: использование `as const` для TS-объектов.

## Boundaries (что НЕ включено)
- Динамическая загрузка данных через API или Fetch во время выполнения.
- Система управления контентом (CMS).
- Поддержка нескольких языков (i18n).
- Вложенная фильтрация продуктов по категориям.

- [x] `tsc --noEmit` проходит без ошибок.

- [x] Все тексты соответствуют тематике "Завод сайтов" (site_factoryall).
- [x] Метрики в кейсах содержат реалистичные бизнес-показатели и соответствуют схеме `{ label, value, unit }`.
- [x] Отсутствуют пустые заглушки (stubs) в данных.
- [x] В `texts.ts` отсутствуют обертки для локализации, строки прописаны напрямую.

### AI_NOTES
- В `texts.ts` следует использовать `as const` для обеспечения неизменяемости статических строк и точного вывода типов литералов.
- Для `cases.json` необходимо создать соответствующий интерфейс `Case` в файле типов (например, `/types/index.ts`), чтобы гарантировать соответствие JSON-структуры ожиданиям компонентов при импорте.
- Пути к изображениям должны строго соответствовать физической структуре папки `public/images/`.
- При добавлении новых продуктов в `products.ts` следить за тем, чтобы массив оставался плоским, согласно A3.
- Метрики (Metrics) должны быть визуально отделены в UI, поэтому важно соблюдать наличие всех трех полей: `label` (название), `value` (число), `unit` (единица измерения, например, "%" или "дней").
