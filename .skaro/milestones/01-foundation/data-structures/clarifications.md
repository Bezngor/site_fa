# Clarifications: data-structures

## Question 1
Какую структуру должны иметь метрики в cases.json для обеспечения единообразия UI?

*Context:* Разные типы метрик (проценты, валюта, сроки) требуют разного форматирования в компоненте CasesSection.

**Options:**
- A) Строгий объект { "label": string, "value": string, "unit": string }
- B) Простой массив строк ["+20% выручки", "2 месяца окупаемости"]
- C) Свободный JSON-объект без жесткой схемы

**Answer:**
Строгий объект { "label": string, "value": string, "unit": string }

## Question 3
Требуется ли поддержка нескольких категорий в ProductLineSection?

*Context:* От этого зависит, будет ли структура products.ts плоским массивом или объектом с ключами-категориями.

**Options:**
- A) Плоский массив (все продукты в один ряд/сетку)
- B) Группировка по категориям { "categoryName": Product[] }
- C) Не нужно, достаточно одного общего списка

**Answer:**
Плоский массив (все продукты в один ряд/сетку)

## Question 4
Каков формат хранения путей к изображениям в JSON/TS файлах?

*Context:* Необходимо определить, использовать ли полные пути от корня public или только имена файлов для next/image.

**Options:**
- A) Абсолютные пути от корня: '/images/cases/case1.webp'
